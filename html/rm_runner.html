AB SDK — ExampleRunner Removal Tracker

Hand-curated, operator-in-the-loop. Not auto-generated.

Why this exists: the previous generate_progress.py output (progress.html) papered over real drift between swagger, source-of-truth models, examples, fixtures, and tests. Generated scripts cannot enforce harmony — they just describe state. This page tracks the migration to plain example scripts where each conversion forces a real reconciliation between the five artefacts.

Goal

  1. Deprecate examples/_runner.py (the ExampleRunner indirection).
  2. Convert each example file in examples/ into a plain SDK script of the form from ab import ABConnectAPI; api = ABConnectAPI(env='staging') followed by direct method calls.
  3. Each conversion must include:
    • a fixture capture path (the script writes tests/fixtures/<Model>.json);
    • a unit test with a mocked HttpClient that locks the wire-level call shape (route, params/body keys);
    • an integration test in tests/integration/ that documents real-world behaviour, including divergences from swagger;
    • any newly discovered forward references between endpoints (output of one feeds input of another) explicitly documented in the example and asserted in tests.
  4. Once an example is converted, the corresponding row below moves from stub to done.

Reference conversion: dashboard

The dashboard group is the first endpoint to be fully converted under the new methodology. Treat it as the template for every other group.

What was produced

  • examples/dashboard.py — plain SDK script. No runner.
  • examples/constants.py::TEST_VIEW_ID — staging view ID sourced from the discovery chain.
  • tests/fixtures/GridViewInfo.json — six representative view rows shaped to swagger GridViewDetails.
  • tests/unit/test_dashboard.py — 15 wire-level tests with a mocked HttpClient.
  • tests/integration/test_dashboard.py — 3 live tests covering the no-params default and the gridviews → get chain.
  • ab/api/models/dashboard.py::GridViewInfo — fields realigned to swagger GridViewDetails (id, name, dataKey, isActive, plus optional sp metadata). Previously had isDefault which is not in swagger.

Operator-in-the-loop discovery sequence

This is the loop the operator follows when capturing or refreshing the dashboard fixture. The same loop becomes the test test_grid_views_chain_to_get.

  1. Call api.dashboard.get_grid_views() — returns List[GridViewInfo].
  2. From that list, pick the entry whose id matches the view you want; ensure is_active is true.
  3. Set TEST_VIEW_ID = <id> in examples/constants.py.
  4. Call api.dashboard.get(view_id=TEST_VIEW_ID, company_id=TEST_COMPANY_ID) — returns DashboardSummary.
  5. Capture tests/fixtures/DashboardSummary.json by running python -m examples.dashboard.

Forward references

ProducerFieldConsumerField
GET /dashboard/gridviews → GridViewInfo id (int) GET /dashboard → DashboardParams view_id (int)

Documented divergences from swagger

Swagger says: GET /dashboard takes viewId and companyId as query parameters. Both are optional in the spec.
Live API actually does: when companyId is omitted, the API substitutes the active user's primary company — so the call succeeds rather than 4xx-ing. The test test_no_params_does_not_fail_and_sends_empty_query (unit) and test_get_no_params_defaults_to_user_primary_company (integration) lock this in.

Canonical example shape

from ab import ABConnectAPI
from examples.constants import TEST_COMPANY_ID, TEST_VIEW_ID

api = ABConnectAPI(env="staging")

# Discovery chain (only needed when TEST_VIEW_ID is unknown):
views = api.dashboard.get_grid_views()  # -> List[GridViewInfo]

# Documented call (swagger contract: both params optional):
summary = api.dashboard.get(view_id=TEST_VIEW_ID, company_id=TEST_COMPANY_ID)

# Documented divergence (no-params is supported in practice):
default_summary = api.dashboard.get()

Per-conversion checklist (operator drives)

For each row above, the operator must:

  • Confirm every method's swagger entry vs. its Route(...) call (path, request_model, response_model, params_model). Patch source-of-truth before continuing.
  • Replace the example file with a plain script: top-level api = ABConnectAPI(env="staging") followed by named call sections.
  • Add any missing constants to examples/constants.py (sourced from real fixtures, not invented).
  • Capture or refresh response fixtures under tests/fixtures/.
  • Add a tests/unit/test_<group>.py with a mocked HttpClient asserting wire-level call shape (route + params/body alias keys).
  • Add a tests/integration/test_<group>.py covering no-params behaviour, the discovery chain, and any documented swagger divergence.
  • Document forward references in the example docstring and assert at least one in tests.
  • Move the row in this file from stub to done.
